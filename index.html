What is a token?

Definition

Token

The smallest unit a model reads and writes. Not a word, not a character — roughly ¾ of a word on average in English.

"Hello" = 1 token. "Unbelievable" = 3 tokens. A typical paragraph ≈ 100 tokens.

Rule of thumb

Rough conversion

100 tokens ≈ 75 words. 1,000 tokens ≈ 750 words ≈ 1.5 pages of text.

A full novel ≈ 100,000–150,000 tokens.

Types

Input vs output tokens

Input = everything you send (prompt, history, system instructions). Output = what the model generates back.

Both count toward the context window. Output tokens cost more to generate.

Context window

Context window

The model's working memory. Input + output combined must stay within the limit. Anything beyond the limit is forgotten.

More context is not always better — accuracy degrades as it fills up (context rot).

Token cost calculation

Cost = (input tokens ÷ 1,000,000) × input rate
+ (output tokens ÷ 1,000,000) × output rate
Example

Haiku 4.5 calculation

Send 10,000 input tokens, get 2,000 output tokens.

Input: 10k ÷ 1M × $0.80 = $0.008
Output: 2k ÷ 1M × $4.00 = $0.008
Total: ~$0.016 (1.6 cents)

Why output costs more

Generation is expensive

The model generates output token by token — computationally heavier than reading input all at once.

Output is typically 3–5× more expensive per token than input.

Cost optimisation

Prompt caching

Repeated context (system prompts, long docs) can be cached. Cache reads cost ~10% of normal input price.

Big saving for apps that reuse the same context across many API calls.

Context rot

Bigger is not always better

All models degrade in accuracy as context fills. Having a 2M window does not mean accuracy holds across all 2M tokens.

Curating what is in context matters as much as window size.

Three levels of AI capability

Traditional

Traditional AI

Classifies, predicts, or labels from structured input. Single-shot. No content generation. No autonomous action.

e.g. spam filters, recommendation engines, image classifiers, fraud detection models

Generative

Generative AI

Creates new content (text, code, image, audio) from a prompt. Single-turn, reactive, stateless by default.

e.g. ChatGPT answering a question, Copilot suggesting code, Midjourney creating an image

Generative agentic

Generative agentic AI

A generative model acts as the reasoning brain of an agent. It plans, takes actions using tools, checks results, and loops until the goal is met.

e.g. Claude Code, AutoGPT, AI research agents, automated coding pipelines

Key risks in agentic AI

Risk

Compounding errors

A wrong step early in the loop can cascade. Unlike a bad answer, a bad action may trigger further bad actions downstream.

Risk

Irreversible actions

Sending an email, deleting a file, or making an API call cannot always be undone. Higher stakes than bad text output.

Risk

Prompt injection

Malicious instructions hidden in data the agent reads (a webpage, a document) can hijack its behaviour mid-task.

Risk

Scope creep

Agent takes actions beyond what was intended. Guardrails, sandboxing, and human-in-the-loop checkpoints are essential.

Three ways to adapt a model's behaviour

Lightest

Prompting

Instructions in the prompt guide the model's behaviour. No training or data pipeline needed. Fast to change, easy to iterate.

Use when: behaviour changes frequently, or you need a quick solution.

Medium

RAG (Retrieval-Augmented Generation)

Fetch relevant documents at runtime and inject them into the prompt. The model's weights are unchanged — it just gets better context.

Use when: answers need current, private, or domain-specific data the model was not trained on.

Heaviest

Fine-tuning

Retrain the model on your own data to change its style, tone, or domain knowledge baked in. Expensive and takes time to set up.

Use when: consistent tone or format is needed at scale and prompting alone is not enough.

How RAG works step by step

Step 1

Index your documents

Convert documents into numerical representations called embeddings. Store them in a vector database (e.g. pgvector, Pinecone, Weaviate).

Step 2

Retrieve at query time

When a user asks a question, convert the question into an embedding and search the vector DB for the most similar document chunks.

Step 3

Inject into prompt

Add the retrieved chunks to the prompt as context. The model answers using both its trained knowledge and the retrieved content.

Step 4

Generate response

The model produces an answer grounded in the retrieved documents. Hallucination risk drops because the answer has a source to draw from.

How an agent is structured

Core

Brain (LLM)

The generative model. Reads the current state and goal, reasons about what to do next, and decides which tool to call.

e.g. Claude Sonnet 4.6, GPT-4o, Gemini 1.5 Pro

Capabilities

Tools

Functions the LLM can call: web search, code execution, file read/write, database queries, API calls.

The agent picks which tool to use and what to pass in. Tools return results back to the model.

State

Memory

Short-term: the context window (what happened so far this session). Long-term: a vector database or external store the agent can read and write.

Without long-term memory, the agent forgets everything when the context window clears.

Execution

The loop

Think → act → observe → think again. The model calls a tool, gets a result, decides what to do next, and repeats until the goal is met or a limit is hit.

This is what makes agents different from a single-turn chat.

Safeguards that matter (especially in civic/government contexts)

Control

Human-in-the-loop

The agent pauses at key decision points for a human to approve before taking an irreversible action.

Control

Guardrails

Rules that constrain what the agent is allowed to do — which tools it can call, which endpoints it can hit, what data it can access.

Control

Sandboxing

The agent runs in an isolated environment where its actions cannot affect production systems without explicit approval.

Control

Audit logs

Every action the agent takes is logged so failures can be traced and reviewed. Essential for accountability in government deployments.

Using AI efficiently — reducing token usage

Prompt design

Be specific, not verbose

A clear, direct prompt gets a better answer in fewer tokens than a long, hedging one. Extra words in your prompt = extra tokens you pay for.

Instead of: "Could you please help me by perhaps summarising this document if that's okay?" — just say: "Summarise this document in 3 bullet points."

Prompt design

Constrain the output

Tell the model how long to be. "In one sentence", "in under 100 words", "as a JSON object with 3 keys" — all reduce output tokens directly.

Unconstrained output is the most common cause of unexpectedly high token bills.

Model selection

Match model to task

Use Haiku for simple, high-volume tasks. Save Sonnet and Opus for tasks that actually need reasoning depth. The quality difference on easy tasks is negligible; the cost difference is 5–20×.

Routing simple queries to Haiku and hard ones to Sonnet/Opus can cut costs dramatically.

Model selection

Use the Batch API

For tasks that do not need an immediate response (data processing, bulk analysis, overnight jobs), the Batch API costs 50% less than the standard API.

Same model, same quality — just async instead of real-time. No justification needed beyond cost.

Context management

Cache repeated context

If you send the same system prompt or large document on every API call, enable prompt caching. Cached tokens cost ~10% of normal input price after the first call.

A 10,000-token system prompt sent 1,000 times: without caching = $30 (at Sonnet rates). With caching = ~$3.

Context management

Trim conversation history

In a long chat, old turns still count as input tokens on every new message. Summarise earlier turns instead of keeping the full transcript, or use a sliding window.

A 50-turn conversation can easily balloon to 30,000+ tokens of input before you type a word.

RAG instead of context stuffing

Retrieve, don't paste

Instead of pasting an entire document into every prompt, use RAG to retrieve only the relevant chunks. Fetching 3 relevant paragraphs costs far less than loading 50 pages.

Context stuffing is the most expensive anti-pattern in production AI applications.

Output format

Ask for structured output

Requesting JSON, a table, or a list is usually more token-efficient than free-form prose. Structured output is also easier to parse programmatically — no regex needed.

Asking for "a JSON object with keys: summary, risk, action" produces tighter output than "please write a paragraph explaining..."

Agentic workflows

Limit agent loop steps

Each loop iteration in an agentic workflow consumes tokens — both the growing history and each tool result fed back in. Set a max step count and summarise intermediate results.

An unbounded agent loop is the fastest way to run up a large API bill unexpectedly.

Agentic workflows

Clear tool results when done

In long agentic tasks, old tool results sitting in context still consume tokens on every step. Use context editing (tool result clearing) to drop results once they are no longer needed.

Supported on Claude Opus 4.6, Sonnet 4.6, and newer models via the API.

The three layers — companies, models, and products

Layer 1

AI lab (the company)

Researches and trains the underlying models. This is where the actual AI is built. The lab owns the model weights and controls how it is used.

e.g. Anthropic, OpenAI, Google DeepMind, Meta AI, Mistral, Cohere

Layer 2

Model (the AI itself)

The trained neural network. A model has a name and version. Multiple products can be built on the same model. The model is not the app — it is the engine.

e.g. Claude Sonnet 4.6, GPT-4o, Gemini 1.5 Pro, Llama 3, Mistral Large

Layer 3

Product (the interface)

The app or tool a person actually uses. A product wraps a model with a UI, memory, tools, and policies. One model can power many products.

e.g. ChatGPT, Claude.ai, Gemini app, GitHub Copilot, Microsoft Copilot

Commonly confused names — cleared up

OpenAI

ChatGPT vs GPT vs OpenAI

OpenAI is the company. GPT (Generative Pre-trained Transformer) is the model family. ChatGPT is the consumer chat product built on GPT models.

Saying "I use ChatGPT" is like saying "I use Gmail" — it tells you the product, not the underlying model.

OpenAI

What is Codex?

Codex was OpenAI's code-specialised model, used to power GitHub Copilot in its early days. It is now retired — GPT-4o and the o-series models handle code far better.

You may still hear "Codex" used loosely to mean AI coding tools in general — but the actual Codex model API was shut down in 2023.

OpenAI

What is the o-series (o1, o3, o4)?

OpenAI's reasoning-focused model line. Separate from the GPT family. "o" models are optimised for step-by-step reasoning and harder problems — similar in spirit to Claude's extended thinking.

o3 and o4-mini are current as of 2025–2026. Not to be confused with GPT-4o (the "o" there means "omni", not the same series).

Google

Gemini vs Bard vs Google AI

Bard was Google's first consumer AI chat product — it was rebranded to Gemini in 2024. The underlying model is also called Gemini. Google DeepMind is the research lab behind it.

Google also has Gemma — a smaller open-source model family separate from the main Gemini line.

Anthropic

Claude vs Claude.ai vs Anthropic

Anthropic is the company. Claude is the model family (Haiku, Sonnet, Opus). Claude.ai is the consumer product — the chat interface you are using right now.

Claude Code is a separate developer tool. The Claude API lets developers build their own products on Claude models.

GitHub Copilot

GitHub Copilot vs Codex vs OpenAI

GitHub Copilot is a product by Microsoft/GitHub. It was originally powered by Codex, then switched to GPT-4o. It is not a model — it is a coding assistant tool built on top of OpenAI models.

Copilot is to OpenAI what Claude.ai is to Anthropic — the product layer, not the model itself.

Microsoft

Microsoft Copilot vs GitHub Copilot

Two different products with similar names. Microsoft Copilot is the general AI assistant built into Windows, Edge, and Office 365. GitHub Copilot is the code-specific tool for developers.

Both are powered by OpenAI models under Microsoft's partnership with OpenAI.

Open source

Open vs closed models

Closed models: weights are private, access is API-only (Claude, GPT-4o, Gemini). Open models: weights are publicly released, you can download and self-host them (Llama 3, Mistral 7B, Gemma).

Open models are popular for privacy-sensitive use cases where you cannot send data to an external API.

The big picture — how it all fits together

Key insight 1

Products ≠ models

ChatGPT, Claude.ai, and Gemini are interfaces. The intelligence comes from the model underneath. A product can switch models — Copilot moved from Codex to GPT-4o without users noticing much.

Key insight 2

One model, many products

The same Claude Sonnet model powers Claude.ai, this artifact, apps built on the Claude API, and enterprise tools. The model is the same — the wrapper, memory, and tools differ.

Key insight 3

Competition is at every layer

Labs compete on model quality. Products compete on UX, integrations, and pricing. A great product on a weaker model can beat a great model with a poor product.

Key insight 4

Open source changes the game

Meta releasing Llama 3 freely means anyone can run a capable model locally. This puts pressure on closed-model providers to justify their pricing through quality, safety, or features.

Local LLM vs cloud LLM — the core difference

Cloud LLM

Runs on the provider's servers

You send your prompt over the internet to a company's data centre. Their infrastructure runs the model and sends the response back. You pay per token or per subscription.

e.g. Claude API, ChatGPT, Gemini API — your data leaves your device and passes through their servers.

Local LLM

Runs on your own hardware

You download the model weights and run them on your own machine or server. No data leaves your environment. No API fees per call. No internet required after setup.

e.g. Llama 3 via Ollama, Mistral 7B, Phi-3 — everything stays on your hardware.

Hardware requirements for local LLMs

Small (1B–7B params)

Entry level

Runs on a modern laptop with 8–16GB RAM. No GPU required — CPU inference is usable though slow. Good for simple Q&A, summarisation, basic coding help.

e.g. Phi-3 Mini, Llama 3 8B, Mistral 7B via Ollama on a MacBook Pro.

Medium (13B–34B params)

Mid-range

Needs 16–32GB RAM, benefits significantly from a GPU. Noticeably better reasoning. Suitable for most knowledge worker tasks locally.

e.g. Llama 3 13B on a Mac Studio M2 or a PC with a mid-range NVIDIA GPU.

Large (70B+ params)

High-end

Needs 64GB+ RAM and a high-end GPU (NVIDIA A100, RTX 4090) or Apple Silicon Mac with 96GB+ unified memory. Approaches cloud model quality.

e.g. Llama 3.1 70B on Mac Studio M3 Ultra (192GB) or a multi-GPU server.

Tools to run local LLMs

Common runtimes

Ollama — easiest setup, runs on Mac/Windows/Linux, one-command model download. LM Studio — GUI-based, good for beginners. llama.cpp — lightweight, runs on almost anything.

Ollama is the recommended starting point for most people: ollama run llama3

How to give a model context and knowledge

Method 1

Context in the prompt

Paste relevant information directly into your message or system prompt. The model reads it as part of the conversation. Simple, no setup. Disappears when the conversation ends.

Works for both local and cloud LLMs. Limited by context window size.

Method 2

RAG (Retrieval-Augmented Generation)

Store your knowledge in a vector database. At query time, fetch only the relevant chunks and inject them into the prompt. Knowledge persists without retraining the model.

Works for both local and cloud. Best for large, frequently updated knowledge bases.

Method 3

Fine-tuning

Retrain the model on your data so the knowledge is baked into the weights permanently. The model learns your domain, style, or facts without needing them in every prompt.

Works for both local and cloud (via API fine-tuning endpoints). More expensive and permanent.

Method 4

Pre-training from scratch

Train an entirely new model on a custom corpus. The model learns everything from your data. Extremely expensive — millions of dollars in compute. Only done by labs and large organisations.

Not a realistic option for most teams. Fine-tuning an existing model is almost always the right approach.

Preventing sensitive information from leaking

Risk 1

Data sent to cloud providers

When you use a cloud LLM (Claude, ChatGPT, Gemini), your prompt is transmitted to and processed on their servers. Even if they have good privacy policies, data leaves your control.

Check your provider's data retention policy. Anthropic, OpenAI, and Google all offer zero data retention options for enterprise plans.

Risk 2

Training data contamination

Some providers may use your conversations to train future models unless you opt out. Consumer tiers (free plans) often have more permissive data use than enterprise tiers.

Always check the terms. Anthropic's API has ZDR (Zero Data Retention) for qualifying enterprise customers. ChatGPT has a training opt-out setting.

Risk 3

Model memorisation

If sensitive data is used in fine-tuning, the model may later reproduce fragments of it when prompted in certain ways — a real risk for PII, passwords, or confidential documents.

Scrub PII and secrets from training data before fine-tuning. Never fine-tune on raw production data.

Risk 4

Prompt injection via shared context

In multi-user or agentic systems, one user's input can be crafted to extract or manipulate another user's context. System prompt leakage is a known attack vector.

Never assume system prompts are secret. Design systems as if the system prompt will be read by an attacker.

Singapore-specific considerations

PDPA

Personal Data Protection Act

Singapore's PDPA requires organisations to protect personal data. Sending personal data of Singapore residents to a cloud AI provider without proper safeguards may constitute a breach.

Ensure your cloud AI provider has data processing agreements and adequate data protection practices before sharing personal data.

Government data

Government classified data

Government data classified as Restricted or above must not be processed on commercial cloud AI services. GovTech provides approved AI tools and platforms for public sector use.

This is directly relevant to the Tf(x) role — understanding data classification in a government context matters.

AI vs chatbot — not the same thing

Traditional chatbot

Rule-based chatbot

Follows a fixed decision tree or script. Responds to specific keywords or button clicks. Cannot handle anything outside its programmed flows. Has no understanding of meaning.

e.g. "Press 1 for billing, press 2 for support." The bot does not understand your question — it matches patterns.

AI chatbot

LLM-powered chatbot

A chatbot interface built on top of a large language model. Understands natural language, handles unexpected questions, generates responses dynamically. The "chatbot" is the interface; the AI is the engine.

e.g. ChatGPT, Claude.ai, Gemini — these are AI chatbots. The conversational format is the interface; the LLM is what gives it intelligence.

AI (broader)

AI is much wider than chatbots

AI includes image generation, voice recognition, fraud detection, recommendation engines, autonomous vehicles, medical imaging, and more. A chatbot is one very narrow application of AI.

Saying "AI" when you mean "chatbot" is like saying "vehicle" when you mean "bicycle." Technically included but wildly undersized.

When to use which

Use a rule-based chatbot when

Predictability matters most

The task is narrow and well-defined. You need guaranteed, auditable responses. Regulatory or compliance requirements mean you cannot have a model improvise. Cost must be minimal.

e.g. booking a GP appointment, checking your CPF balance, resetting a password — scripted flows are safer and cheaper.

Use an AI chatbot when

Flexibility matters most

Users ask unpredictable questions. The knowledge base is large or changes frequently. You need to handle nuance, context, and follow-up questions naturally.

e.g. internal knowledge base Q&A, customer support for complex products, educational tutoring tools like what Tf(x) builds.

Hybrid approach

Often the best architecture

Use rule-based flows for high-stakes, predictable paths (payment, authentication). Use an LLM for open-ended discovery and support. Route between them based on intent detection.

Most mature enterprise AI deployments use this pattern — deterministic for critical paths, generative for the rest.

How to use agentic AI effectively

Principle 1

Give goals, not step-by-step instructions

Agentic AI is designed to figure out the steps itself. Tell it what outcome you want, not how to achieve it. Over-specifying each step removes the benefit of using an agent.

Instead of: "First search X, then open Y, then copy Z..." — just say: "Find me the three most relevant papers on topic X and summarise their key findings."

Principle 2

Define the scope and boundaries clearly

Tell the agent what it is allowed to do and — equally important — what it is not. An agent with vague permissions will either do too little (asks constantly) or too much (takes actions you did not intend).

e.g. "You can read any file in this folder but do not write, delete, or send anything without asking me first."

Principle 3

Break large tasks into checkpointed subtasks

For complex multi-hour tasks, design the workflow so the agent reports progress at key milestones rather than running silently to completion. This limits blast radius if something goes wrong.

Think of it like a contractor relationship: daily check-ins beat discovering a problem after a week of work.

Principle 4

Verify outputs before they become inputs

In a multi-step workflow, a wrong output in step 2 becomes the wrong input for step 3. Build human review or automated validation between steps for high-stakes pipelines.

Especially important for agentic code generation — always review generated code before running it in production.

Principle 5

Match the model to the task in the loop

Not every step in an agent loop needs the most powerful model. Use a cheap fast model (Haiku) for simple tool-call decisions and a strong model (Sonnet/Opus) for the core reasoning steps.

This is called "model routing" — the right model per step, not one model for everything.

Principle 6

Set a step limit and time budget

Always cap how many loop iterations an agent can take. Without a limit, a confused agent can loop indefinitely — burning tokens and potentially making repeated mistakes.

A reasonable default: 20–30 steps for complex tasks. Alert a human if the limit is reached rather than failing silently.

Principle 7

Provide the right tools — no more, no less

Only give the agent tools it actually needs for the task. An agent with access to a database delete function and an email sender is far more dangerous than one that can only read and summarise.

Principle of least privilege — same as in system security, applied to AI agents.

Principle 8

Log everything

Every tool call, every decision, every intermediate result should be logged. When something goes wrong (and it will), you need to trace exactly which step caused the problem.

Observability is the most underrated part of building reliable agentic systems.

Practical prompt patterns for agentic tasks

Pattern 1

Goal + constraints + output format

State what you want, what limits apply, and what the result should look like. This three-part structure prevents most ambiguity.

"Research the top 5 open-source vector databases [goal]. Do not access any paywalled sources [constraint]. Return a comparison table with columns: name, licence, language, key strength [output format]."

Pattern 2

Explicit stop conditions

Tell the agent when to stop and when to ask rather than guess. Agents that are unsure will often keep trying — sometimes making things worse.

"If you cannot find a reliable source after 3 searches, stop and tell me what you found so far rather than continuing."

Pattern 3

Persona and role framing

For complex agentic tasks, framing the agent's role helps it make better decisions about what to prioritise and what to ignore.

"You are a careful senior engineer reviewing code for security issues. Be conservative — flag anything suspicious rather than assuming it is safe."

Pattern 4

Chain of thought for key decisions

For decisions with significant consequences, ask the agent to reason out loud before acting. This surfaces errors before they become actions.

"Before taking any action, write out your reasoning for why that action is the right next step."
